4. Mainnet deploy sequence

Point of no return. Steps below broadcast real transactions on Base mainnet (chain 8453). They are irreversible. You'll be prompted twice for confirmation.

4a. Run the orchestrator

HOUSE_BRAWLERS_COUNT=10 \
HOUSE_KEEPER_ADDRESS=0x<keeper-address> \
npm run deploy:mainnet

Phases:

  1. Preflight — tools, env, RPC, balance, build, test
  2. Forge deploy — broadcasts BRAWL, Brawlers, Duel, Graveyard, MintDrop, Marketplace in one script. Deploy.s.sol also wires duel.setMarketplace() so listed brawlers can't be sent into duels, and (when HOUSE_BRAWLERS_COUNT is set) mints N house brawlers to the deployer, flags them as isHouseBrawler on-chain, and transfers them to HOUSE_KEEPER_ADDRESS.
  3. Parse broadcast — extracts addresses (MARKETPLACE_ADDRESS included)
  4. Update .env — writes addresses into .env.base-mainnet
  5. Mint King — sends King NFT (#2001) to deployer
  6. Update Vercel env — production env vars set (incl. NEXT_PUBLIC_MARKETPLACE_ADDRESS + NEXT_PUBLIC_HOUSE_BRAWLER_IDS when house count > 0)
  7. Vercel production deploy — frontend points at mainnet
  8. Smoke test — read-only checks against deployed contracts

v11 change: Marketplace folded into the main forge script. v12 change: HOUSE_BRAWLERS_COUNT env now seeds house fighters at deploy time so day-1 players always have someone to brawl. House-flagged brawlers in the 1..100 founder range get NO founder discount / freebie / airdrop.

6. Enable trading + anti-sniper window

6a. Whitelist game contracts (final pre-launch state)

# Game contracts that must move BRAWL during gameplay:
cast send $BRAWL_ADDRESS "setWhitelistBulk(address[],bool)" \
  "[$DUEL_ADDRESS,$GRAVEYARD_ADDRESS,$MINTDROP_ADDRESS]" true \
  --private-key $DEPLOYER_KEY --rpc-url $MAINNET_RPC

6b. Flip trading on (THE LAUNCH MOMENT)

cast send $BRAWL_ADDRESS "enableTrading()" \
  --private-key $DEPLOYER_KEY --rpc-url $MAINNET_RPC

This is the launch tx. Within seconds:

  • BRAWL is tradeable on the LP
  • The 1-block bot trap is armed (sniper bots that frontrun this tx get auto-blacklisted)
  • Max-tx 0.5% (500 BRAWL) and max-wallet 1% (1k BRAWL) limits apply
  • Fire X launch thread tweet 1/12 NOW

6c. Watch the first 24-48h for snipers

Use Basescan transfers panel + DexScreener trades feed. If you spot:

  • A wallet that bought within seconds of launch and immediately tried to dump → blacklist it
  • Coordinated wallets buying small amounts under max-tx → blacklist all of them
cast send $BRAWL_ADDRESS \
  "blacklist(address,string)" \
  $BAD_WALLET "sniper" \
  --private-key $DEPLOYER_KEY --rpc-url $MAINNET_RPC

6d. Freeze the rarity table after the dev-mint window

v11 ships Brawlers.freezeRarity(): once the dev wallet has taken its mints, this owner-only call locks the rarity table so subsequent buyers see no further skip-rare swaps. The shuffle is deterministic from masterSeed, so anyone can re-derive the post-shuffle hash off-chain and confirm initialRarityHash matches.

# Verify the on-chain table still matches the canonical shuffle:
cast call $BRAWLERS_ADDRESS "initialRarityHash()(bytes32)" --rpc-url $MAINNET_RPC
cast call $BRAWLERS_ADDRESS "rarityHash()(bytes32)"        --rpc-url $MAINNET_RPC
# If they differ, dev-skip swaps have fired (expected if the dev wallet has minted).

# Lock it in (idempotent):
cast send $BRAWLERS_ADDRESS "freezeRarity()" \
  --private-key $DEPLOYER_KEY --rpc-url $MAINNET_RPC

After this tx, public buyers can be sure the rarity they pull is exactly what's at their tokenId in the now-frozen table. Call this before posting the rarity sheet to Discord/X.

7. Contract verification on Basescan

Etherscan multichain API key already stored in .env.base-mainnet. Run for each contract.

source .env.base-mainnet

forge verify-contract $BRAWL_ADDRESS contracts/BRAWL.sol:BRAWL \
  --chain-id 8453 \
  --etherscan-api-key $ETHERSCAN_API_KEY \
  --constructor-args $(cast abi-encode "constructor(address,address)" \
    $DEPLOYER_ADDRESS $DEPLOYER_ADDRESS) \
  --watch

forge verify-contract $BRAWLERS_ADDRESS contracts/Brawlers.sol:Brawlers \
  --chain-id 8453 \
  --etherscan-api-key $ETHERSCAN_API_KEY \
  --constructor-args $(cast abi-encode "constructor(string,uint256,address)" \
    "https://baseicbrawlers.com/api/token/" \
    "0x2a" \
    $DEPLOYER_ADDRESS) \
  --watch

# Repeat for Duel, Graveyard, MintDrop, Marketplace with their respective
# constructor args. Run once per contract; --watch polls until verified.

All contracts verified ⇒ green checkmark on Basescan. People reading the source see real Solidity, not bytecode. Critical for trust.

10. Fight-cost keeper deployment

Keeper bot lives at marketing/keeper/. Deploy on TrueNAS as a sibling Dockge stack.

10a. Create keeper EOA + fund

Generate a fresh EOA (e.g. cast wallet new). Fund it with ~0.005 ETH on Base. This wallet's only job is to call setFightEconomics every few minutes.

10b. Transfer Duel contract ownership to the keeper

The current owner is the deployer. After mainnet stabilises (~24h), transfer Duel ownership to the keeper EOA:

cast send $DUEL_ADDRESS \
  "transferOwnership(address)" \
  $KEEPER_ADDRESS \
  --private-key $DEPLOYER_KEY \
  --rpc-url $MAINNET_RPC

If you'd rather not transfer ownership, leave it with deployer and have keeper run as a notifier-only (DRY_RUN=true). Manual rebalance from deployer when needed.

10c. Fill .env on TrueNAS

SCP marketing/keeper/.env.example → fill KEEPER_PRIVATE_KEY, DUEL_ADDRESS, BRAWL_ADDRESS, BRAWL_PAIR_ADDRESS, DEV_TREASURY. Set DRY_RUN=true for first 24h.

10d. Deploy via Dockge

scp -r marketing/keeper/ truenas-discord:/mnt/.ix-apps/app_mounts/dockge/stacks/baseic-keeper/
ssh truenas-discord docker compose \
  -f /mnt/.ix-apps/app_mounts/dockge/stacks/baseic-keeper/docker-compose.yml \

13. Post-launch operations (first 7 days)

  • Discord: respond to every #help ticket within 4h waking hours. AutoMod handles the obvious scams.
  • Telegram: bot bots running (welcome / raid / leaderboard); manually monitor for impersonators.
  • Bots: keep an eye on the keeper logs. Adjust DELTA_THRESHOLD if updates are too chatty or too slow.
  • Mint: monitor tier transitions. When tier 1 (1-50) sells out, post a "tier 1 SOLD OUT, tier 2 active at $25" tweet. Repeat for each tier.

  • Marketplace: watch for floor sweep / wash trading. Manual investigation if anything looks off.

14. Renounce sequence (24-48h after launch)

After trading has stabilised + obvious snipers blacklisted, surrender control of BRAWL:

14a. Lift the launch limits

cast send $BRAWL_ADDRESS "liftLimits()" \
  --private-key $DEPLOYER_KEY --rpc-url $MAINNET_RPC

One-way. Removes max-tx and max-wallet caps. Trading is now unrestricted.

14b. Renounce BRAWL ownership

cast send $BRAWL_ADDRESS "renounceOwnership()" \
  --private-key $DEPLOYER_KEY --rpc-url $MAINNET_RPC

Whitelist/blacklist/limits frozen forever. BRAWL contract becomes truly immutable.

Game contracts (Duel/MintDrop/Marketplace/Graveyard) STAY dev-controlled for tuning. No rug-able functions in those — only economics setters.

If something is on fire after launch.
IncidentResponse
Snipers draining LPManual blacklist of confirmed wallets via BRAWL.blacklist(addr, reason). If catastrophic, you can still call this — owner not yet renounced.

Smart contract bug surfacesGame contracts have pause(). BRAWL has no pause (would be a censorship lever). For BRAWL bugs after renounce, only the community / forks can react.
